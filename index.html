Swift Basics

Variables · Optionals · Functions · Closures
What it is

Swift is Apple's compiled, type-safe programming language introduced in 2014 as a modern replacement for Objective-C. It is used to build apps for iOS, macOS, watchOS, and tvOS. Swift's standout features are optionals (values that may or may not exist), type inference (the compiler figures out types from context), closures (self-contained blocks of code you can pass around), and clean syntax that reduces boilerplate. Every iOS project starts here — models, logic, views, and services are all written in Swift.

Code Snippet
swift 
// Constants and variables from AppUser model
let appName: String = "Chilechilo Cards"    // immutable
var coinBalance: Int = 0                     // mutable coin count

// Type inference — compiler deduces the type
var lastPackTime = Date()                    // Date
var canOpenFree = false                      // Bool

// Optionals — a value that might be nil (card image URLs)
var cardImageURL: String? = nil
cardImageURL = "https://storage.chilechilo.com/cards/gobernador.jpg"

// Safe unwrapping — guard lets you exit early
func displayCard(_ name: String?) -> String {
    guard let name = name else {
        return "Unknown Chilean figure"
    }
    return "Tarjeta: \(name)"
}

// nil coalescing — provide a fallback
let displayName = cardImageURL ?? "placeholder_card.png"

// Closure — filter rarity cards
let allCardRarities = ["bronze", "silver", "gold", "platinum"]
let rarePulls = allCardRarities.filter { $0 == "gold" || $0 == "platinum" }
print(rarePulls)   // ["gold", "platinum"]

// async/await — modern concurrency in Firestore
func fetchAllChileanCards() async throws -> [Card] {
    let snapshot = try await db.collection("cards").getDocuments()
    return snapshot.documents.compactMap { try? $0.data(as: Card.self) }
}
🃏
Real Use Case

Chilechilo Cards is entirely written in Swift. Optionals handle missing image URLs and optional user data fields, closures power pack-opening animations and card filtering, and async/await manages Firestore queries seamlessly. The AppUser model uses optionals for lastPackOpened to check if a free pack timer is valid.

02

OOP — Classes, Structs, Enums & Protocols

Value types · Reference types · Named cases · Contracts
What it is

Swift has four building blocks for organizing data and behavior. Structs are value types — copied when assigned or passed, making them thread-safe and predictable by default. Classes are reference types — all variables point to the same object in memory, ideal for shared mutable state. Enums define a fixed, exhaustive set of named cases with optional associated values. Protocols define a contract — a required set of properties and methods — that any type can adopt, enabling polymorphism and dependency injection without rigid class hierarchies.

Code Snippet
swift 
// Struct — the core Card model from Firestore
import Foundation
import FirebaseFirestore

struct Card: Identifiable, Codable {
    @DocumentID var id: String?
    var name: String
    var rarity: String = "bronze"
    var imageURL: String = "placeholder"
}

// Struct for user-owned cards (count tracking)
struct UserCard: Identifiable, Codable {
    var id: String
    var count: Int
}

// Enum — rarity tiers with associated colors
enum Rarity: String, CaseIterable {
    case bronze = "Bronze"
    case silver = "Silver"
    case gold = "Gold"
    case platinum = "Platinum"
}

// Class — AppUser model for authentication state
struct AppUser: Codable {
    var email: String
    var coins: Int
    var lastPackOpened: Date
}

// Usage in CollectionViewModel
let chilechilo_gobernador = Card(
    id: "1",
    name: "Gobernador de Chilechilo",
    rarity: "gold",
    imageURL: "https://storage.com/gobernador.jpg"
)

let owned = UserCard(id: "1", count: 2)  // User has 2 copies
print("Owns \(owned.count) × \(chilechilo_gobernador.name)")  // Owns 2 × Gobernador

// Struct is value type — each collection view gets its own copy
var displayCard = chilechilo_gobernador
displayCard.name = "Foil " + displayCard.name  // Doesn't affect original
print(chilechilo_gobernador.name)  // Still "Gobernador de Chilechilo"
🏗️
Real Use Case

In Chilechilo Cards, Card is a struct with @DocumentID (safe to copy inside SwiftUI cells and decodable from Firestore), UserCard tracks owned counts by ID, and Rarity is an enum for filtering rarity tiers. AppUser is a simple Codable struct synced from Firestore. Identifiable and Codable protocols are required by SwiftUI's List and Firestore's automatic JSON decoding.

03

SwiftUI — Building the Chilechilo UI

Declarative UI · Stacks · State · Composition
What it is

SwiftUI is Apple's declarative UI framework, released in 2019. Instead of imperatively creating and configuring views one by one (like UIKit), you declare what the UI looks like given a certain state — and SwiftUI re-renders the affected parts automatically when that state changes. Layouts are built by composing stacks: VStack (vertical), HStack (horizontal), and ZStack (layered). Modifiers chain off any view to customize font, color, padding, shadow, animation, and more — each modifier returns a new, wrapped view so they compose cleanly.

Code Snippet
swift 
import SwiftUI

struct CardRowView: View {
    let card: Card
    @State private var isFavorited = false

    var body: some View {
        VStack(alignment: .leading, spacing: 12) {

            // Card image from Firestore URL
            ZStack(alignment: .topTrailing) {
                AsyncImage(url: URL(string: card.imageURL)) { image in
                    image.resizable().scaledToFill()
                } placeholder: {
                    Color.gray.opacity(0.3)
                }
                .frame(height: 200)
                .clipped()
                .cornerRadius(12)

                // Rarity badge (bronze/silver/gold/platinum)
                Text(card.rarity.uppercased())
                    .font(.caption2.weight(.bold))
                    .foregroundStyle(.white)
                    .padding(.horizontal, 8).padding(.vertical, 4)
                    .background(rarityColor(card.rarity))
                    .cornerRadius(6)
                    .padding(8)
            }

            // Card name and rarity
            VStack(alignment: .leading, spacing: 4) {
                Text(card.name)
                    .font(.headline.weight(.semibold))
                    .foregroundStyle(.primary)
                Text(card.rarity)
                    .font(.caption)
                    .foregroundStyle(.secondary)
            }

            // Favorite toggle
            HStack {
                Spacer()
                Button {
                    withAnimation(.easeInOut(duration: 0.2)) {
                        isFavorited.toggle()
                    }
                } label: {
                    Image(systemName: isFavorited ? "star.fill" : "star")
                        .foregroundStyle(isFavorited ? .yellow : .gray)
                }
            }
        }
        .padding(12)
        .background(Color(.systemBackground))
        .cornerRadius(12)
        .shadow(color: .black.opacity(0.08), radius: 8, y: 2)
    }

    func rarityColor(_ rarity: String) -> Color {
        switch rarity.lowercased() {
        case "bronze": return .brown
        case "silver": return .gray
        case "gold": return .yellow
        case "platinum": return .cyan
        default: return .blue
        }
    }
    }
}
📱
Real Use Case

Every screen in Chilechilo Cards is built with SwiftUI. The collection view uses a List of CardRowViews fetched from Firestore. AsyncImage loads card artwork URLs directly from cloud storage. All styling through modifier chains with dynamic rarity-based colors. @State on isFavorited causes the star button to re-render automatically on each tap.

MVVM + Firebase

Architecture · Firestore sync · Observable state
What it is

Model-View-ViewModel (MVVM) is an architecture pattern that separates code into three layers. The Model holds raw data. The ViewModel contains business logic, fetches from Firestore, and transforms data for display. The View simply renders what the ViewModel exposes — no logic lives there. In iOS 17+, @ObservableObject + @Published automatically notifies SwiftUI when state changes.

Code Snippet
swift 
import SwiftUI
import Combine
import FirebaseFirestore

// ── CollectionViewModel — loads cards & owned status ──────
@MainActor
class CollectionViewModel: ObservableObject {
    @Published var allCards: [Card] = []
    @Published var ownedIds: Set = []
    @Published var ownedCounts: [String: Int] = [:]
    
    private let manager = FirestoreManager()

    // Derived state — computed from cards
    var cardCount: Int { ownedIds.count }
    var isEmpty: Bool { ownedIds.isEmpty }

    func load() async {
        do {
            allCards = try await manager.fetchAllCards()
            let owned = try await manager.fetchOwnedCards()
            ownedIds = Set(owned.map { $0.id })
            ownedCounts = Dictionary(
                uniqueKeysWithValues: owned.map { ($0.id, $0.count) }
            )
        } catch {
            print("Error loading collection: \(error)")
        }
    }

    func addCard(_ card: Card) {
        guard let cardId = card.id else { return }
        ownedIds.insert(cardId)
        ownedCounts[cardId, default: 0] += 1
    }

    func removeCard(withId id: String) {
        ownedIds.remove(id)
        ownedCounts.removeValue(forKey: id)
    }
}

// ── View — only renders, zero business logic ───────────
struct BinderView: View {
    // @State with @Observable (no @StateObject needed)
    @State private var vm = BinderViewModel()

    var body: some View {
        NavigationStack {
            Group {
                if vm.isEmpty {
                    ContentUnavailableView(
                        "Your binder is empty",
                        systemImage: "square.stack.3d.up",
                        description: Text("Scan physical cards or browse the database to add them")
                    )
                } else {
                    List {
                        ForEach(vm.cards) { card in
                            HStack {
                                Text(card.name)
                                Spacer()
                                Text("$\(card.marketPrice, specifier: "%.2f")")
                                    .foregroundStyle(.secondary)
                            }
                            .swipeActions {
                                Button("Delete", role: .destructive) {
                                    vm.remove(card)
                                }
                            }
                        }

                        Section {
                            HStack {
                                Text("Total Portfolio Value").bold()
                                Spacer()
                                Text("$\(vm.totalValue, specifier: "%.2f")").bold()
                            }
                        }
                    }
                }
            }
            .navigationTitle("My Binder (\(vm.cardCount))")
            .toolbar {
                ToolbarItem(placement: .bottomBar) {
                    Button("Sync Collection") { Task { await vm.syncWithCloud() } }
                        .disabled(vm.isEmpty)
                        .buttonStyle(.borderedProminent)
                }
            }
            .overlay { if vm.isLoading { ProgressView() } }
        }
    }
}
🔁
Real Use Case

CollectionViewModel handles all Firestore queries, owned card tracking, and collection statistics. The View layer only renders and dispatches events. This separation allows testing card-fetching logic independently of UI presentation, and makes swapping the backend trivial.

Navigation — NavigationStack & TabView

Push-pop hierarchy · Tab bars · Deep linking
What it is

NavigationStack (iOS 16+) manages a push/pop hierarchy of views. The user drills in by tapping a NavigationLink and goes back with the system back button. It uses a type-safe, value-based API: you attach a navigationDestination modifier that maps a value type to a destination view. TabView creates a tab bar at the bottom of the screen — each tab is an independent root view with its own navigation stack. Together, these two patterns cover nearly every navigation need in a standard iOS app.

swift

Lists & Tables

Lazy rows · Swipe actions · Sections · Search
What it is

List is SwiftUI's primary scrollable container for displaying rows of data. It renders lazily — only visible rows are created, making it efficient even with thousands of items. List supports swipe actions, pull-to-refresh, editing mode, multi-selection, and sections. The .searchable modifier adds a system search bar that filters the data source in real time. For data-grid-style layouts on iPad and macOS, Table provides sortable columns with a header row. List and Table cover virtually every data-display scenario in iOS apps.

Code Snippet
swift 
import SwiftUI

struct CardBinderListView: View {
    @State private var allCards: [TradingCard] = TradingCard.samples
    @State private var searchText = ""

    var filteredCards: [TradingCard] {
        searchText.isEmpty
            ? allCards

                $0.name.localizedCaseInsensitiveContains(searchText)
              }

        Dictionary(grouping: filteredCards) { $0.type }
    }

    var body: some View {
        List {
            ForEach(CardType.allCases, id: \.self) { typeKey in
                if let sections = groupedByTypes[typeKey], !sections.isEmpty {
                    Section(typeKey.rawValue) {
                        ForEach(sections) { card in
                            CardRowView(card: card)
                                // Swipe left — destructive delete from binder
                                .swipeActions(edge: .trailing) {
                                    Button("Delete", role: .destructive) {
                                        deleteCard(card)
                                    }
                                }
                                // Swipe right — quick add to wishlist
                                .swipeActions(edge: .leading) {
                                    Button {
                                        addToWishlist(card)
                                    } label: {
                                        Label("Wishlist", systemImage: "heart.badge.plus")

                                    .tint(.green)

                        }

Firestore & Real-Time Data

Codable · async/await · Document queries · Auth sync
What it is

Firebase Firestore is a cloud NoSQL database accessed directly from iOS apps via the Firebase SDK. Collections hold documents (JSON-like objects) that are automatically decoded into Swift Codable types. Queries are written in a type-safe builder API and return results asynchronously via async/await. Security rules define read/write access. Unlike REST APIs where you manually sync, Firestore can push real-time updates to listeners, keeping the UI automatically in sync with the backend.

Code Snippet
swift 
import SwiftUI
import FirebaseFirestore
import FirebaseAuth

// ── 1. Firestore manager — centralized data access ────
@MainActor
class FirestoreManager: ObservableObject {
    private let db = Firestore.firestore()

    // GET — fetch all Chilean cards from Firestore collection
    func fetchAllCards() async throws -> [Card] {
        let snapshot = try await db.collection("cards").getDocuments()
        print("📦 Found \(snapshot.documents.count) cards")
        return snapshot.documents.compactMap { try? $0.data(as: Card.self) }
    }

    // GET — fetch user's owned cards and counts
    func fetchOwnedCards() async throws -> [UserCard] {
        guard let uid = Auth.auth().currentUser?.uid else { return [] }
        let snapshot = try await db.collection("userCards")
            .document(uid)
            .collection("owned")
            .getDocuments()
        return try snapshot.documents.compactMap { try $0.data(as: UserCard.self) }
    }
}

// ── 2. Authentication manager with Firestore ──────────
class AuthManager: ObservableObject {
    @Published var isLoggedIn = false
    private let db = Firestore.firestore()

    func register(email: String, password: String) async throws {
        let result = try await Auth.auth().createUser(withEmail: email, password: password)
        try await createUserDocument(uid: result.user.uid, email: email)
        await MainActor.run { isLoggedIn = true }
    }

    private func createUserDocument(uid: String, email: String) async throws {
        let data: [String: Any] = [
            "email": email,
            "coins": 0,
            "lastPackOpened": Timestamp(date: Date(timeIntervalSince1970: 0))
        ]
        try await db.collection("users").document(uid).setData(data)
        print("✅ User document created in Firestore")
    }
}

// ── 3. View that loads Firestore data ────────────────
struct CollectionView: View {
    @StateObject private var colVM = CollectionViewModel()

    var body: some View {
        List {
            ForEach(colVM.allCards) { card in
                Text(card.name)
            }
        }
        .task {
            await colVM.load()  // Fetch from Firestore
        }
    }
}
🌐
Real Use Case

Chilechilo Cards uses Firebase Firestore for all data persistence. FirestoreManager handles querying the global card registry and user-owned cards. AuthManager automatically creates a Firestore user document on signup with initial coin balance. All data is decoded directly into Codable Swift models (Card, UserCard, AppUser).

Maps — MapKit

Interactive maps · Markers · Annotations · Routes
What it is

MapKit is Apple's framework for embedding interactive maps in iOS apps. The SwiftUI Map view (iOS 17+ API) accepts a MapCameraPosition binding that controls the viewport, and displays Marker and Annotation views at geographic coordinates. You can show the user's real-time location with UserAnnotation, draw route overlays with MapPolyline, and respond to map tap and item-selection events. MapKit tiles are powered by Apple Maps and automatically adapt to light and dark mode.

Code Snippet
swift 
import SwiftUI

Camera & QR Reader

AVFoundation · AVCaptureSession · UIViewRepresentable
What it is

iOS camera access is managed by AVFoundation. An AVCaptureSession wires together inputs (the camera device) and outputs (still images, video, or metadata). For QR reading, you add an AVCaptureMetadataOutput — the framework fires a delegate method every time a QR code or barcode is detected in the live camera frame. Because AVFoundation is a UIKit API, it must be wrapped in a UIViewRepresentable to bridge into SwiftUI. The camera permission key NSCameraUsageDescription must be added to Info.plist.

Code Snippet
swift 
import SwiftUI
import AVFoundation

// ── UIViewRepresentable bridges AVFoundation → SwiftUI ─
struct CardScannerView: UIViewRepresentable {
    var onTokenDetected: (String) -> Void

    // Coordinator acts as the AVFoundation delegate
    class Coordinator: NSObject, AVCaptureMetadataOutputObjectsDelegate {
        let targetView: CardScannerView
        init(_ viewInstance: CardScannerView) { self.targetView = viewInstance }

        func metadataOutput(
            _ output: AVCaptureMetadataOutput,
            didOutput objects: [AVMetadataObject],
            from connection: AVCaptureConnection
        ) {
            guard
                let rawObj = objects.first as? AVMetadataMachineReadableCodeObject,
                let valueString = rawObj.stringValue
            else { return }

            // Ensure UI actions run safely on the primary loop thread
            DispatchQueue.main.async {
                self.targetView.onTokenDetected(valueString)
            }
        }
    }

    func makeCoordinator() -> Coordinator { Coordinator(self) }

    func makeUIView(context: Context) -> UIView {
        let view          = UIView(frame: .zero)
        let captureStream = AVCaptureSession()

        guard
            let cameraSource = AVCaptureDevice.default(for: .video),
            let captureInput = try? AVCaptureDeviceInput(device: cameraSource),
            captureStream.canAddInput(captureInput)
        else { return view }

        captureStream.addInput(captureInput)

        let metadataPipeline = AVCaptureMetadataOutput()
        captureStream.addOutput(metadataPipeline)
        metadataPipeline.setMetadataObjectsDelegate(context.coordinator, queue: .main)
        metadataPipeline.metadataObjectTypes = [.qr, .code128, .ean13]

        // Video configuration layer context linkage
        let renderingLayer = AVCaptureVideoPreviewLayer(session: captureStream)
        renderingLayer.videoGravity = .resizeAspectFill
        renderingLayer.frame = UIScreen.main.bounds
        view.layer.addSublayer(renderingLayer)

        DispatchQueue.global(qos: .userInitiated).async {
            captureStream.startRunning()
        }
        return view
    }

    func updateUIView(_ uiView: UIView, context: Context) {}
}

// ── SwiftUI screen that uses the scanner ──────────────

CoreLocation

GPS · Permissions · CLLocation · Distance
What it is

CoreLocation gives iOS apps access to the device's geographic position using GPS, Wi-Fi triangulation, and cellular data. Before receiving any location, you must request user permission — either when in use (only while the app is open) or always (including when backgrounded). Updates are delivered via CLLocationManagerDelegate callbacks. You can compute the distance between two CLLocation objects with a single method call, or set up geofences (CLCircularRegion) that fire events when the user enters or exits an area. The required Info.plist key is NSLocationWhenInUseUsageDescription.

Code Snippet
swift 
import CoreLocation
import SwiftUI

// ── Observable wrapper around CLLocationManager ────────
@Observable
class UserLocationManager: NSObject, CLLocationManagerDelegate {
    private let coreLocationService = CLLocationManager()
    var lastRecordedLocation: CLLocation?
    var permissionStatus: CLAuthorizationStatus = .notDetermined

    override init() {
        super.init()
        coreLocationService.delegate = self
        coreLocationService.desiredAccuracy = kCLLocationAccuracyNearestTenMeters
    }

    func requestDevicePermission() {
        coreLocationService.requestWhenInUseAuthorization()
    }

    // Delegate — called when permission changes
    func locationManagerDidChangeAuthorization(_ manager: CLLocationManager) {
        permissionStatus = manager.authorizationStatus
        if permissionStatus == .authorizedWhenInUse || permissionStatus == .authorizedAlways {
            coreLocationService.startUpdatingLocation()
        }
    }

    // Delegate — called with new GPS fix
    func locationManager(_ manager: CLLocationManager, didUpdateLocations positions: [CLLocation]) {
        lastRecordedLocation = positions.last
    }

    func locationManager(_ manager: CLLocationManager, didFailWithError error: Error) {
        print("GPS Stream Error: \(error.localizedDescription)")
    }

    // Utility: distance calculation metric output in meters
    func measureMetresToTarget(targetCoords: CLLocationCoordinate2D) -> Double? {
        guard let currentAnchor = lastRecordedLocation else { return nil }
        let targetSpot = CLLocation(latitude: targetCoords.latitude, longitude: targetCoords.longitude)
        return currentAnchor.distance(from: targetSpot)
    }
}

// ── View: show nearest trade meetup store ─────────────
struct ClosestStoreView: View {
    @State private var locationService = UserLocationManager()
    let retailNetworks = CardShopLocation.samples

    var closestShop: CardShopLocation? {
        retailNetworks.min {
            (locationService.measureMetresToTarget(targetCoords: $0.coordinates) ?? .infinity) <
            (locationService.measureMetresToTarget(targetCoords: $1.coordinates) ?? .infinity)
        }
    }

    var body: some View {
        VStack(spacing: 24) {
            if locationService.permissionStatus == .notDetermined {
                Button("Authorize Location Tracking") {
                    locationService.requestDevicePermission()
                }
                .buttonStyle(.borderedProminent)
            } else if let shop = closestShop,
                      let rangeDist = locationService.measureMetresToTarget(targetCoords: shop.coordinates) {
                VStack(spacing: 6) {
                    Text("Closest Local Trading Spot").font(.caption).foregroundStyle(.secondary)
                    Text(shop.shopName).font(.title2.bold())
                    Text(String(format: "%.0f meters from current location", rangeDist))
                        .foregroundStyle(.secondary)
                }
            } else {
                ProgressView("Calibrating device GPS metrics…")
            }
        }
        .onAppear { locationService.requestDevicePermission() }
        .padding()
    }
}
