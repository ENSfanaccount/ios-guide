01

Swift Basics

Variables · Optionals · Functions · Closures
What it is

Swift is Apple's compiled, type-safe programming language introduced in 2014. It is the primary language for iOS, macOS, watchOS, and tvOS. Its standout features are optionals (values that may or may not exist, written as Type?), type inference (the compiler deduces types from context), closures (anonymous functions you assign and pass around), and a clean syntax that eliminates Objective-C's verbosity. Every file in an iOS project is Swift — models, services, ViewModels, and views all start here.

Code Snippet
swift
// Constants and variables
let packCost: Int = 150      // immutable — pack price never changes
var coins: Int = 0           // mutable — player balance updates

// Type inference
var cardName = "Jalapeño Ace"    // inferred as String
var isOwned  = false             // inferred as Bool

// Optional — @DocumentID in Card produces a String?
var cardId: String? = nil
cardId = "bronze_01"

// Guard — safe early exit when optional is nil
func logCard(_ id: String?) {
    guard let id = id else {
        print("Card has no Firestore ID yet")
        return
    }
    print("Processing card: \(id)")
}

// nil coalescing — provide a fallback value
let display = cardId ?? "unknown"   // "bronze_01"

// Closure — used when filtering cards by rarity
let allCards: [Card] = []
let goldOnly = allCards.filter { $0.rarity == "gold" }

// Switch — used in rarityColor() and drawCard() in PackManager
func coinValue(for rarity: String) -> Int {
    switch rarity {
    case "bronze":  return 10
    case "silver":  return 30
    case "gold":    return 100
    case "diamond": return 300
    default:        return 5
    }
}

// async/await — all Firestore calls use this pattern
func fetchCards() async throws -> [Card] {
    // ... URLSession or Firestore SDK call
    return []
}
🃏
Real Use Case — Chilechilo Cards

In Chilechilo Cards, Card.id is String? because Firestore's @DocumentID is only populated after fetching — guard unwraps it safely before any Firestore write. The coinValue(for:) function mirrors processDuplicate(rarity:) in PackManager.swift, and closures power the .filter and .map calls throughout every ViewModel.

02

OOP — Classes, Structs, Enums & Protocols

Value types · Reference types · Named cases · Contracts
What it is

Swift provides four building blocks for organizing data and behavior. Structs are value types — copied when assigned or passed, making them safe and predictable (ideal for data models). Classes are reference types — all variables point to the same object in memory, making them right for shared, mutable state like managers and services. Enums define a fixed set of named cases with optional associated values. Protocols define contracts — required properties and methods — that any type can conform to, enabling polymorphism without inheritance.

Code Snippet
swift
import FirebaseFirestore

// ── Struct — value type, used for all data models ─────
// From Card.swift
struct Card: Identifiable, Codable {
    @DocumentID var id: String?   // Firestore document ID
    var name: String
    var rarity: String = "bronze"
    var imageURL: String = "placeholder"
}

// From UserCard.swift
struct UserCard: Identifiable, Codable {
    var id: String
    var count: Int
}

// From AppUser.swift
struct AppUser: Codable {
    var email: String
    var coins: Int
    var lastPackOpened: Date
}

// ── Enum — could be used to type-safe rarities ────────
// (Currently stored as String in the project;
//  an enum makes the compiler catch typos at build time)
enum Rarity: String, CaseIterable {
    case bronze  = "bronze"
    case silver  = "silver"
    case gold    = "gold"
    case diamond = "diamond"

    var coinValue: Int {
        switch self {
        case .bronze:  return 10
        case .silver:  return 30
        case .gold:    return 100
        case .diamond: return 300
        }
    }
}

// ── Class — reference type, used for managers/services ─
// Mirrors AuthManager.swift
class AuthManager: ObservableObject {
    @Published var isLoggedIn = false

    func login(email: String, password: String) async throws {
        // Firebase Auth call
    }
    func logout() {
        // Auth.auth().signOut()
        isLoggedIn = false
    }
}

// ── Protocol — Identifiable and Codable are protocols ──
// Conforming to Identifiable lets SwiftUI's ForEach work
// Conforming to Codable lets JSONDecoder / Firestore decode
protocol CardDisplayable {
    var name: String { get }
    var rarity: String { get }
}

// Card already satisfies this automatically
extension Card: CardDisplayable {}

Real Use Case — Chilechilo Cards

Card, UserCard, and AppUser are all structs — safe to copy into SwiftUI views. AuthManager, FirestoreManager, and PackManager are classes — shared instances that hold live Firebase connections and mutable state. Conforming Card to Identifiable and Codable was required by both SwiftUI's ForEach and Firestore's data(as:) decoder.

03

SwiftUI — Views, Layouts & Modifiers

Declarative UI · Stacks · ZStack · Modifiers
What it is

SwiftUI is Apple's declarative UI framework (iOS 13+). Instead of imperatively creating and configuring views step by step, you declare what the UI looks like for a given state — SwiftUI handles rendering and automatically re-renders when state changes. Layouts are built by composing stacks: VStack (vertical), HStack (horizontal), and ZStack (layered / overlapping). Modifiers chain off any view to change its appearance, behavior, or layout — each modifier returns a new wrapped view, so they compose cleanly and read like a recipe.

Code Snippet
swift
import SwiftUI

// Directly from CollectionView.swift — CardThumbnail
struct CardThumbnail: View {
    let card: Card
    let owned: Bool
    let count: Int

    var body: some View {
        // ZStack layers content on top of each other
        ZStack(alignment: .topTrailing) {

            // Background — rarity color if owned, gray if locked
            RoundedRectangle(cornerRadius: 12)
                .fill(owned
                      ? rarityColor(card.rarity)   // bronze/silver/gold/diamond
                      : Color.gray.opacity(0.3))
                .frame(height: 140)

            if owned {
                // Card image from Assets
                Image(card.imageURL)
                    .resizable()
                    .scaledToFill()
                    .frame(width: 90, height: 130)
                    .clipShape(RoundedRectangle(cornerRadius: 12))

                // Duplicate count badge — only shown when count > 1
                if count > 1 {
                    Text("x\(count)")
                        .font(.caption.bold())
                        .foregroundColor(.white)
                        .padding(4)
                        .background(Color.black.opacity(0.6))
                        .clipShape(RoundedRectangle(cornerRadius: 6))
                        .padding(4)   // positions the badge inside the ZStack corner
                }
            } else {
                // Locked card — show padlock
                Image(systemName: "lock.fill")
                    .foregroundColor(.gray)
                    .font(.title)
            }
        }
    }
}

// rarityColor() is a global helper in RarityColor.swift
// It returns a SwiftUI Color based on the rarity string
func rarityColor(_ rarity: String) -> Color {
    switch rarity {
    case "bronze":  return Color(red: 0.8, green: 0.5, blue: 0.2)
    case "silver":  return Color(red: 0.7, green: 0.7, blue: 0.75)
    case "gold":    return Color(red: 0.9, green: 0.75, blue: 0.1)
    case "diamond": return Color(red: 0.4, green: 0.8, blue: 1.0)
    default:        return .gray
    }
}
🎴
Real Use Case — Chilechilo Cards

CardThumbnail in CollectionView.swift uses a ZStack to layer the card image on top of a colored background. The rarityColor() helper drives the background fill so bronze, silver, gold, and diamond cards each have a distinct color. Unowned cards show a gray fill with a lock icon — all controlled purely through modifiers with no imperative code.

MVVM + @Observable

Architecture · @ObservableObject · @Observable · Reactive state
What it is

Model-View-ViewModel (MVVM) separates code into three layers: the Model holds raw data, the ViewModel contains business logic and exposes state for the View, and the View only renders. In SwiftUI there are two ways to make a class observable. The older ObservableObject + @Published pattern (iOS 14+) re-renders every subscriber when any @Published property changes. The newer @Observable macro (iOS 17+) is more granular — each View only re-renders for the exact properties it reads. Both patterns appear in this project.

Code Snippet
swift
import SwiftUI
import Observation   // needed for @Observable

// ── Pattern A: @ObservableObject (used in PackViewModel) ─
@MainActor
class PackViewModel: ObservableObject {
    @Published var coins = 0
    @Published var canOpenFree = false
    @Published var timeRemaining: TimeInterval = 0
    @Published var drawnCards: [Card] = []
    @Published var showReveal = false

    private let manager = PackManager()

    func openPack() {
        drawnCards = manager.openPack(allCards: allCards)
        showReveal = true   // triggers the sheet in HomeView
    }

    var formattedTime: String {
        let h = Int(timeRemaining) / 3600
        let m = (Int(timeRemaining) % 3600) / 60
        let s = Int(timeRemaining) % 60
        return String(format: "%02d:%02d:%02d", h, m, s)
    }
}

// ── Pattern B: @Observable (used in LocationViewModel) ──
// iOS 17+ — more granular, no @Published needed
@Observable
class LocationViewModel: NSObject, CLLocationManagerDelegate {
    var latitude: Double  = 0.0
    var longitude: Double = 0.0
    var authorizationStatus: CLAuthorizationStatus = .notDetermined
    var isUpdating = false

    private let manager = CLLocationManager()

    override init() {
        super.init()
        manager.delegate = self
        manager.desiredAccuracy = kCLLocationAccuracyBest
    }

    func requestPermission() {
        manager.requestWhenInUseAuthorization()
    }
}

// ── How they're used in Views ─────────────────────────
struct HomeView: View {
    // @StateObject for ObservableObject
    @StateObject private var packVM = PackViewModel()

    var body: some View {
        Text("Coins: \(packVM.coins)")
    }
}

struct LocationView: View {
    // @State for @Observable (no @StateObject needed)
    @State private var vm = LocationViewModel()

    var body: some View {
        Text("Lat: \(vm.latitude)")
    }
}
🔁
Real Use Case — Chilechilo Cards

The project uses both patterns. PackViewModel, CollectionViewModel, and AuthViewModel use the older ObservableObject + @Published approach (broader iOS compatibility). LocationViewModel uses the newer @Observable macro — notice it is declared with @State in LocationView, not @StateObject.

Navigation — NavigationStack & TabView

Push-pop · Tab bars · Sheets · navigationDestination
What it is

NavigationStack (iOS 16+) manages a push/pop view hierarchy. The user drills into a child view via a NavigationLink and goes back with the back button. The type-safe navigationDestination(for:) modifier declares which view to show for a given value type, decoupling navigation triggers from destinations. TabView creates a tab bar at the bottom of the screen, giving users access to top-level sections of the app. .sheet presents a modal overlay that the user can dismiss by pulling down — perfect for temporary flows like opening a card pack.

Code Snippet
swift
import SwiftUI

// ── HomeView — TabView with 4 tabs (from HomeView.swift) ─
struct HomeView: View {
    @StateObject private var packVM  = PackViewModel()
    @StateObject private var colVM   = CollectionViewModel()

    var body: some View {
        TabView {
            packTab
                .tabItem { Label("Inicio",    systemImage: "house.fill") }

            CollectionView()
                .tabItem { Label("Colección", systemImage: "square.grid.2x2.fill") }

            ProfileView()
                .tabItem { Label("Perfil",    systemImage: "person.fill") }

            LocationView()
                .tabItem { Label("Ubicación", systemImage: "location.fill") }
        }
        // Sheet — shown when a pack is opened
        .sheet(isPresented: $packVM.showReveal, onDismiss: {
            Task {
                await colVM.load()

Lists & Tables

LazyVGrid · Adaptive columns · ForEach · task
What it is

List is SwiftUI's lazy scrollable container for displaying rows. LazyVGrid and LazyHGrid extend the concept to grids — only visible cells are rendered, keeping memory use low even with hundreds of items. GridItem(.adaptive(minimum:)) creates as many columns as fit the available width at the given minimum size, making the grid automatically responsive across iPhone and iPad. ForEach iterates over a data collection and renders a view per item. The .task modifier kicks off async data loading when the view first appears.

Code Snippet
swift
import SwiftUI

// ── CollectionView — full grid from the project ───────
struct CollectionView: View {
    @StateObject private var vm = CollectionViewModel()

    // .adaptive creates as many columns as fit at ≥100 pt wide
    let columns = [GridItem(.adaptive(minimum: 100))]

    var body: some View {
        NavigationStack {
            ScrollView {
                LazyVGrid(columns: columns, spacing: 16) {
                    ForEach(vm.allCards) { card in
                        CardThumbnail(
                            card: card,
                            owned: vm.ownedIds.contains(card.id ?? ""),
                            count: vm.ownedCounts[card.id ?? ""] ?? 0
                        )
                    }
                }
                .padding()
            }
            .navigationTitle("Mi colección")
            // .task runs once when the view appears, cancels on disappear
            .task { await vm.load() }
        }
    }
}

// ── CollectionViewModel — loads data into the grid ───
@MainActor
class CollectionViewModel: ObservableObject {
    @Published var allCards: [Card]         = []
    @Published var ownedIds: Set<String>    = []
    @Published var ownedCounts: [String: Int] = [:]

    private let manager = FirestoreManager()

    func load() async {
        do {
            // Fetch all 60+ cards from Firestore
            allCards = try await manager.fetchAllCards()

            // Fetch which ones the logged-in user owns
            let owned = try await manager.fetchOwnedCards()

            // Build a Set for O(1) lookup in CardThumbnail
            ownedIds    = Set(owned.map { $0.id })
            ownedCounts = Dictionary(uniqueKeysWithValues:
                            owned.map { ($0.id, $0.count) })
        } catch {
            print("Error cargando colección: \(error)")
        }
    }
}

// ── Tip: filtering the grid by rarity ────────────────
// Adding a filter segmented control over the grid

                }

}

CollectionView uses a LazyVGrid with GridItem(.adaptive(minimum: 100)) to display all cards in a responsive grid. HTTP & JSON — Codable + Firestore

Codable · @DocumentID · Firestore SDK · Real-time listeners
What it is

Most iOS apps talk to a backend over HTTP. Apple's URLSession handles raw HTTP requests with async/await, and the Codable protocol maps JSON to Swift types automatically via JSONDecoder. Firebase Firestore is a NoSQL cloud database whose iOS SDK wraps HTTP internally — it exposes the same Codable integration through data(as:), so your model structs decode from Firestore documents the same way they would from JSON. Firestore also supports real-time listeners that push updates to the app whenever a document changes in the cloud.

Code Snippet
swift
import FirebaseFirestore
import FirebaseAuth

// ── Card.swift — Codable + @DocumentID ───────────────
// @DocumentID maps the Firestore document ID into the struct
struct Card: Identifiable, Codable {
    @DocumentID var id: String?   // filled by Firestore SDK on decode
    var name:     String
    var rarity:   String = "bronze"
    var imageURL: String = "placeholder"
}

// ── FirestoreManager.swift — fetch methods ────────────
@MainActor
class FirestoreManager: ObservableObject {
    private let db = Firestore.firestore()

    // GET all cards from the "cards" collection
    func fetchAllCards() async throws -> [Card] {
        let snapshot = try await db.collection("cards").getDocuments()
        // data(as:) uses Codable — same as JSONDecoder
        return snapshot.documents.compactMap { try? $0.data(as: Card.self) }
    }

    // GET cards owned by the current user (subcollection)
    func fetchOwnedCards() async throws -> [UserCard] {
        guard let uid = Auth.auth().currentUser?.uid else { return [] }
        let snapshot = try await db
            .collection("userCards")
            .document(uid)
            .collection("owned")
            .getDocuments()
        return try snapshot.documents.compactMap { try $0.data(as: UserCard.self) }
    }
}

// ── PackViewModel.swift — real-time listener ──────────
// Firestore can push live updates using addSnapshotListener
private func fetchUserData() {
    guard let uid = Auth.auth().currentUser?.uid else { return }

    Firestore.firestore()
        .collection("users").document(uid)
        .addSnapshotListener { [weak self] snapshot, _ in
            guard let data = snapshot?.data() else { return }

            let coins = data["coins"] as? Int ?? 0
            let last  = (data["lastPackOpened"] as? Timestamp)?.dateValue()
                         ?? Date(timeIntervalSince1970: 0)

            Task { @MainActor [weak self] in
                self?.coins       = coins
                self?.canOpenFree = self?.manager.canOpenFree(lastOpened: last) ?? false
            }
        }
}

// ── PackManager — write operations ───────────────────
// After opening a pack, save results back to Firestore
func savePackResult(cards: [Card], ownedIds: Set<String>) async throws -> Int {
    guard let uid = Auth.auth().currentUser?.uid else { return 0 }
    let ownedRef = db.collection("userCards").document(uid).collection("owned")
    var coinsEarned = 0

    for card in cards {
        let cardId = card.id ?? ""
        if ownedIds.contains(cardId) {
            coinsEarned += processDuplicate(rarity: card.rarity)
        } else {
            // Write new owned card document
            try await ownedRef.document(cardId).setData(["id": cardId, "count": 1])
        }
    }
    try await db.collection("users").document(uid).updateData([
        "coins": FieldValue.increment(Int64(coinsEarned))
    ])
    return coinsEarned
}
☁️
Real Use Case — Chilechilo Cards

FirestoreManager fetches both the full card catalog and the user's subcollection in two async calls — Codable via data(as:) handles decoding automatically. PackViewModel attaches a real-time listener to the user document so the coin counter and timer update instantly across sessions without manual refresh. After a pack is opened, PackManager writes new owned cards and updates the coin balance atomically with FieldValue.increment.

n8n & Web Services

Workflow automation · Webhooks · Excel as database · URLSession
What it is

n8n is an open-source workflow automation platform — a visual tool where you chain together "nodes" (triggers, logic, services) without writing backend code. Each workflow can expose an HTTP webhook endpoint that an iOS app can call with URLSession. For our food ordering project, n8n acted as the entire backend: one workflow read the menu from an Excel spreadsheet and returned it as JSON; another received an order from the app and appended a new row to the Excel sheet. The iOS side just called those webhook URLs with standard Codable models.

Code Snippet
swift
import Foundation

// ── Models — map the JSON that n8n returns ────────────
struct MenuItem: Codable, Identifiable {
    let id:       String
    let name:     String
    let price:    Double
    let category: String
}

struct OrderRequest: Codable {
    let tableNumber: Int
    let items:       [String]   // array of MenuItem IDs
}

struct OrderResponse: Codable {
    let orderId: String
    let total:   Double
    let status:  String   // "received", "preparing", "ready"
}

// ── n8n service — calls webhook endpoints ─────────────
struct N8nService {
    // The n8n instance URL (self-hosted or cloud)
    static let base = "https://n8n.example.com/webhook"

    // GET /menu — n8n reads the Excel sheet, returns JSON array
    static func fetchMenu() async throws -> [MenuItem] {
        let url = URL(string: "\(base)/menu")!
        let (data, response) = try await URLSession.shared.data(from: url)
        guard (response as? HTTPURLResponse)?.statusCode == 200
        else { throw URLError(.badServerResponse) }
        return try JSONDecoder().decode([MenuItem].self, from: data)
    }

    // POST /order — n8n appends a row to Excel, returns receipt
    static func placeOrder(_ order: OrderRequest) async throws -> OrderResponse {
        var req = URLRequest(url: URL(string: "\(base)/order")!)
        req.httpMethod = "POST"
        req.setValue("application/json", forHTTPHeaderField: "Content-Type")
        req.httpBody = try JSONEncoder().encode(order)

        let (data, _) = try await URLSession.shared.data(for: req)
        return try JSONDecoder().decode(OrderResponse.self, from: data)
    }
}

// ── ViewModel that uses the service ───────────────────
@MainActor
class MenuViewModel: ObservableObject {
    @Published var items: [MenuItem] = []
    @Published var isLoading = false
    @Published var errorMessage: String?

    func loadMenu() async {
        isLoading = true
        defer { isLoading = false }
        do {
            items = try await N8nService.fetchMenu()
        } catch {
            errorMessage = error.localizedDescription
        }
    }
}

// ── View ──────────────────────────────────────────────
struct MenuView: View {
    @StateObject private var vm = MenuViewModel()

    var body: some View {
        List(vm.items) { item in
            HStack {
                Text(item.name)
                Spacer()
                Text("$\(item.price, specifier: "%.2f")")
                    .foregroundStyle(.secondary)
            }
        }
        .overlay { if vm.isLoading { ProgressView() } }
        .task { await vm.loadMenu() }
        .navigationTitle("Menú")
    }
}
⚙️
Real Use Case — Food Ordering App

For the food ordering project, n8n hosted two workflows: a GET webhook that read the menu from a shared Excel file and returned each row as a JSON object, and a POST webhook that received the order JSON and appended a new row to the orders sheet. The iOS app called both endpoints with URLSession — n8n handled all the data logic, so the app had no backend server of its own.

TestFlight & App Store

Distribution · Archive · App Store Connect · Info.plist
What it is

TestFlight is Apple's official beta testing platform. Before an app reaches the App Store, you upload a build to App Store Connect and distribute it to testers via TestFlight — either internal testers (up to 100 people on your team) or external testers (up to 10,000 via a public link) who install the app on their real device without it being publicly listed. The process requires a valid Bundle Identifier, a signing certificate, and an Info.plist with all required privacy usage descriptions. For the SwiftUILearning app, TestFlight let classmates and the professor install and test the app on their own iPhones before final submission.

Code Snippet

UIKit Fundamentals

UIView · UIViewController · UIViewRepresentable · UIApplication
What it is

UIKit is Apple's original imperative UI framework, present since iOS 2 (2008). You construct views by creating objects, setting properties, and adding subviews manually — the opposite of SwiftUI's declarative approach. Even in a SwiftUI app, UIKit surfaces in several places: UIApplication.shared exposes app-level operations (opening URLs, accessing the status bar), UIViewRepresentable bridges any UIView into SwiftUI so you can use UIKit-only components, and many Apple frameworks (AVFoundation, MapKit overlays, WebKit) still ship UIKit APIs that need wrapping before they can appear in a SwiftUI view.

Code Snippet

import SwiftUI
import UIKit
import WebKit

// ── 1. UIApplication — used in LocationView.swift ────
// When location is denied, this button opens iOS Settings.
// UIApplication.openSettingsURLString is a UIKit constant.
Button("Abrir Ajustes") {
    if let url = URL(string: UIApplication.openSettingsURLString) {
        UIApplication.shared.open(url)   // UIKit API inside SwiftUI
    }
}
// This exact code appears in LocationView.swift for the
// .denied / .restricted authorization case.

// ── 2. UIViewRepresentable — bridge UIKit → SwiftUI ──
// Wraps WKWebView (UIKit) so it renders inside a SwiftUI view.
struct WebView: UIViewRepresentable {
    let url: URL

    // makeUIView: called ONCE to create the UIKit view
    func makeUIView(context: Context) -> WKWebView {
        let wv = WKWebView()
        wv.load(URLRequest(url: url))
        return wv
    }

    // updateUIView: called when SwiftUI state changes
    func updateUIView(_ uiView: WKWebView, context: Context) {
        // reload or update the WKWebView if url changes
    }
}

// Usage inside a SwiftUI View
struct TermsView: View {
    var body: some View {
        WebView(url: URL(string: "https://example.com/terms")!)
            .ignoresSafeArea()
    }
}

// ── 3. UIViewControllerRepresentable ─────────────────
// Same idea for full UIViewControllers — e.g. PHPickerViewController
// to let the user pick photos from their camera roll.
struct PhotoPicker: UIViewControllerRepresentable {
    var onPick: (UIImage) -> Void

    func makeUIViewController(context: Context)
        -> PHPickerViewController {
        var config = PHPickerConfiguration()
        config.filter = .images
        let picker = PHPickerViewController(configuration: config)
        picker.delegate = context.coordinator
        return picker
    }

    func updateUIViewController(_ uiViewController: PHPickerViewController,
                                context: Context) {}

    func makeCoordinator() -> Coordinator { Coordinator(self) }

    class Coordinator: NSObject, PHPickerViewControllerDelegate {
        let parent: PhotoPicker
        init(_ p: PhotoPicker) { parent = p }

        func picker(_ picker: PHPickerViewController,
                    didFinishPicking results: [PHPickerResult]) {
            picker.dismiss(animated: true)
            results.first?.itemProvider.loadObject(ofClass: UIImage.self) { obj, _ in
                if let image = obj as? UIImage {
                    DispatchQueue.main.async { self.parent.onPick(image) }
                }
            }
        }
    }
}
